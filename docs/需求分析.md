# 图片压缩工具需求分析文档

> **文档版本：** V1.0
> **文档性质：** 功能需求分析
> **目标读者：** 产品经理、项目经理、开发团队

---

## 一、项目概述

### 1.1 项目背景

随着数字化内容爆发式增长，图片已成为日常办公、网络传输、内容创作中不可或缺的部分。然而，高分辨率图片往往体积庞大，带来存储空间浪费、网页加载缓慢、传输效率低下等问题。用户对图片压缩的需求日益普遍，尤其在以下场景中表现突出：

- 设计师需要批量压缩作品集以便于客户预览和邮件传输
- 运营人员需要压缩产品图片以优化电商平台上传速度
- 普通用户在社交平台发帖前压缩照片以节省流量
- 开发者在交付项目时压缩素材包以降低分发体积

当前市面上已有的在线压缩工具大多存在以下痛点：强制注册登录才能使用完整功能、压缩质量不可控、批量处理能力弱、无法直观对比压缩效果。本工具旨在为用户提供一个**免费、无门槛、功能聚焦、体验流畅**的图片压缩解决方案。

### 1.2 项目目标

开发一款面向电脑端的本地图片压缩工具，用户无需注册登录即可使用。工具支持图片上传、灵活压缩参数配置、压缩前后效果对比，并最终以可执行程序（exe/app）的形式交付，使工具可以独立运行在任何电脑上，无需安装任何运行时环境。

### 1.3 目标用户

| 用户群体 | 典型使用场景 | 核心诉求 |
|---|---|---|
| 设计师 | 压缩作品集/交付文件 | 保持视觉质量、批量处理 |
| 运营/市场人员 | 压缩产品图/活动素材 | 操作简单、批量处理 |
| 开发者 | 压缩项目素材/文档配图 | 支持多种格式、可控压缩率 |
| 普通用户 | 压缩照片/社交图片 | 操作直观、效果可见 |

### 1.4 术语定义

| 术语 | 定义 |
|---|---|
| 源文件 | 用户上传的原始图片文件 |
| 压缩文件 | 经过压缩处理后的输出图片文件 |
| 压缩率 | 压缩前与压缩后文件大小的比值，反映压缩程度 |
| 质量参数 | 控制压缩输出质量的数值（通常为百分比或等级） |
| 批量压缩 | 一次性对多张图片进行压缩处理 |
| 对比视图 | 将压缩前后的图片并排或叠加展示，供用户比较效果 |

## 二、功能需求

### 2.1 图片上传

#### 2.1.1 单图上传

- 用户通过点击上传按钮，从本地文件系统选择一张图片文件
- 支持的图片格式：`JPG/JPEG`、`PNG`、`WebP`、`BMP`、`GIF`（静态帧）
- 文件选择对话框应支持常见的图片文件类型过滤
- 上传后应立即在界面上显示图片预览
- 显示图片基本信息：文件名、原始尺寸（宽×高像素）、文件大小、格式

#### 2.1.2 批量上传

- 用户可通过多选（按住 Ctrl/Cmd 键）或选择文件夹的方式一次性上传多张图片
- 支持批量上传的图片总数上限建议不少于 **50 张**
- 批量上传后，界面应展示图片列表，列出每张图片的名称、大小和状态
- 支持从列表中移除单张或清空全部已上传的图片

#### 2.1.3 拖拽上传

- 用户可以将图片文件直接从操作系统的文件管理器中**拖拽**到工具窗口内以上传
- 支持拖拽单张和多张图片
- 拖拽进入窗口时，应有明确的视觉反馈（如高亮边框或提示文案）

#### 2.1.4 上传限制与校验

- 单个文件大小上限：**100 MB**（超出时给出友好提示）
- 不支持的格式应给出明确错误提示，并告知支持哪些格式
- 上传过程中应显示上传进度（如有多个文件）

### 2.2 图片压缩

#### 2.2.1 压缩参数配置

用户应能够对压缩行为进行灵活配置，具体参数如下：

- **质量等级 / 质量滑块**
  - 提供 0~100 的滑动条或预设等级（低/中/高/极高）
  - 默认为中等质量（约 70~80）
  - 实时预览调整后的预估文件大小
  - 可针对不同图片设置不同质量

- **输出格式选择**
  - 保持原格式
  - 统一转换为 JPG
  - 统一转换为 PNG
  - 统一转换为 WebP（若工具支持 WebP 编码）

- **输出尺寸调整（可选，列为可选功能）**
  - 缩放比例：如 50%、75%、100%
  - 指定最大宽度/高度（等比缩放）
  - 注意：尺寸调整与质量压缩为独立的可叠加操作

#### 2.2.2 压缩执行

- 提供明确的"开始压缩"按钮触发压缩操作
- 单张图片压缩时，实时显示压缩进度条
- 批量压缩时，显示当前处理序号和总进度（如 `3/20`）
- 压缩过程中允许取消操作
- 压缩完成后，显示每张图片的压缩结果（压缩后大小、节省比例、压缩后格式）

#### 2.2.3 压缩引擎要求

- 压缩过程在本地执行，不上传图片到任何服务器
- 压缩过程不应导致图片出现明显的视觉质量下降（尤其在高画质区间）
- 对于已经高度压缩的 JPG 图片，应避免二次压缩带来不必要的伪影

### 2.3 压缩前后对比

#### 2.3.1 对比视图

- **并排对比**：将原图与压缩后的图片并排放置，左右或上下布局
- **同步缩放**：在对比视图中，拖动缩放或滚动时，原图和压缩图应同步缩放和平移，确保视觉对齐
- **切换对比**：提供切换按钮或滑块，可以快速在"只看原图"/"只看压缩图"/"对比视图"之间切换

#### 2.3.2 对比信息展示

- 清晰展示以下数据对比项：
  - 文件大小：原文件大小 vs 压缩后文件大小（以 KB/MB 为单位）
  - 节省幅度：以百分比形式显示（如"节省了 68%"）
  - 图片尺寸：宽×高（像素）
  - 图片格式：原格式 vs 输出格式
  - 压缩质量参数

#### 2.3.3 批量对比

- 当处理多张图片时，提供列表视图展示每张图片的压缩摘要
- 用户可点击列表中的任意条目，切换到该图片的详细对比视图

### 2.4 图片导出与保存

#### 2.4.1 单张保存

- 提供"保存"或"下载"按钮，将当前压缩后的图片保存到本地
- 默认保存路径建议为原图所在目录的子文件夹（如 `原图名称_compressed`）
- 首次保存时弹出系统文件保存对话框，允许用户自定义保存路径和文件名
- 保存成功后，显示保存路径的提示

#### 2.4.2 批量导出

- 提供"导出全部"功能，将所有压缩后的图片批量保存到指定目录
- 导出时允许用户选择目标文件夹
- 导出完成后，显示导出结果摘要（成功多少张、失败多少张）

#### 2.4.3 文件名命名规则

- 默认命名规则：`原文件名_compressed.扩展名`
- 若同一目录下已存在同名文件，应自动添加序号：`原文件名_compressed_1.扩展名`

### 2.5 压缩历史记录

#### 2.5.1 历史记录存储

- 工具应自动保存每次压缩操作的完整记录，包括：
  - 原图文件名、路径、尺寸、格式、原始大小
  - 压缩后文件名、路径、尺寸、格式、压缩后大小
  - 使用的压缩参数（质量等级、输出格式、是否启用尺寸调整）
  - 压缩时间戳
  - 节省比例（%）
- 历史记录存储在本地，无需上传至任何服务器
- 支持至少保存最近 **100 条**历史记录，超出时自动清除最早的记录
- 工具关闭后重新打开，历史记录应从本地存储（JSON 文件或 SQLite）中恢复

#### 2.5.2 历史记录查看

- 提供独立的"历史记录"面板或抽屉（Drawer），可通过工具栏或快捷键唤起
- 历史记录以时间倒序排列（最新在前），显示每条记录的：
  - 原图缩略图
  - 文件名
  - 压缩时间（相对时间：如"3 分钟前"、"昨天 14:32"）
  - 节省比例（以颜色区分：绿色≥50%、黄色 20%~50%、红色 <20%）
  - 原始大小 → 压缩后大小
- 支持按文件名关键词搜索历史记录
- 支持按时间段筛选（今天 / 近 7 天 / 近 30 天 / 全部）

#### 2.5.3 历史记录操作

- 点击任意历史条目，可直接打开该图片的压缩结果（若文件仍存在）
- 提供"重新压缩"功能：读取历史记录中的参数设置，快速将同一张图片以相同参数重新压缩
- 提供"应用到其他图片"功能：将历史记录中的压缩参数（质量、格式、尺寸）一键应用到当前上传的图片
- 提供手动删除单条历史记录的功能
- 提供"清空全部历史"功能，需二次确认

#### 2.5.4 历史记录数据持久化

- 历史记录以 JSON 格式存储在应用数据目录（如 `~/Library/Application Support/ai-images-tools/history.json`）
- 每次压缩完成后自动追加记录到本地文件
- 应用启动时自动加载历史记录
- 支持历史记录的导入与导出（JSON 格式），方便备份和迁移

## 三、用户界面与交互需求

### 3.1 整体布局

- 工具为**单窗口应用**，所有功能在同一个界面内完成
- 主要布局分区（建议）：
  - **顶部工具栏**：标题、设置入口、关闭按钮
  - **左侧/顶部上传区**：拖拽上传区、上传按钮、图片列表
  - **中央主工作区**：图片预览、对比视图
  - **底部/右侧控制面板**：压缩参数配置、导出按钮

### 3.2 交互体验要求

- 所有按钮和可交互元素应在视觉上可识别（光标变化、悬停状态）
- 操作反馈及时：点击按钮后应在视觉上有即时响应
- 图片加载和压缩处理中应显示加载状态（loading 动画或进度条），避免用户误以为界面卡死
- 错误提示应清晰、友好，不使用纯技术性的错误信息
- 支持键盘操作（Tab 键切换焦点，Enter 键确认）

### 3.3 主题与适配

- 默认采用**浅色主题**，界面简洁清爽
- 窗口支持自由调整大小，最小尺寸应保证核心功能可完整展示
- 主工作区中的图片应支持自适应缩放以适应窗口大小

## 四、数据与性能需求

### 4.1 数据处理

- 所有图片处理均在本地完成，**不涉及任何网络上传**
- 工具运行时不应收集或传输任何用户图片数据
- 内存占用：单张图片处理时，工具整体内存占用不超过 **500 MB**
- 临时文件：压缩过程中产生的临时文件应在操作完成后自动清理

### 4.2 性能基准

| 操作 | 性能要求 |
|---|---|
| 10 MB JPG 图片压缩 | 处理时间 ≤ 5 秒 |
| 50 张图片批量压缩 | 每张平均处理时间 ≤ 10 秒 |
| 界面启动时间 | 从点击图标到界面可操作 ≤ 3 秒 |
| 图片切换/对比响应 | 操作响应 ≤ 200 ms |

## 五、交付形态需求

### 5.1 可执行程序

- 最终交付物为独立的**可执行程序**（Windows exe 或 macOS app）
- 用户下载/获取该程序后，双击即可运行，无需安装 Python、Node.js、Java 等任何运行时环境
- 打包时应将所有依赖项和运行环境完整打包进可执行文件中

### 5.2 兼容性

- 目标操作系统：Windows 10/11（64位）、macOS 11+
- 可执行程序应可在没有网络连接的环境下正常运行

## 六、非功能需求（辅助说明）

> 以下为辅助性的质量要求，供产品设计和开发团队参考。

### 6.1 易用性

- 用户无需阅读使用说明即可完成基本的压缩操作
- 上传、压缩、对比、保存四大核心操作路径应尽量短（不超过 3 步）

### 6.2 可靠性

- 工具不应因单张图片处理失败而崩溃或无法继续处理后续图片
- 压缩失败时应给出明确提示，说明可能原因（如文件损坏、格式不支持）

### 6.3 可扩展性（供参考，不作强制要求）

- 未来可能扩展的功能：更多图片格式支持（如 TIFF、RAW）、水印添加、图片裁剪、历史记录等

## 七、功能优先级

根据需求的必要性和实现复杂度，对功能需求进行优先级划分：

### P0（核心功能，必须实现）

- 图片上传（单图 + 拖拽）
- 压缩参数配置（质量滑块、输出格式）
- 压缩执行与进度展示
- 压缩前后对比视图
- 单张图片保存

### P1（重要功能，建议实现）

- 批量上传（多选）
- 批量导出
- 批量压缩与列表管理
- 输出尺寸调整

### P2（增强功能，可选实现）

- 同步缩放对比
- 文件名自定义规则
- 界面主题切换
- **压缩历史记录（P1 建议实现，P2 可选）**

## 八、需求确认清单

以下条目供需求确认时逐项核对：

- [ ] 是否需要注册/登录功能？ → **否**
- [ ] 图片是否需要上传到服务器？ → **否，本地处理**
- [ ] 是否需要支持批量操作？ → **是，建议支持**
- [ ] 是否需要对比视图？ → **是，核心功能**

- [ ] 是否需要多语言支持？ → **暂不需要，首版仅中文/英文**
- [ ] 是否需要打印或导出报告功能？ → **否**
