> **文档版本：** V1.0
> **技术栈：** Electron + Vue 3 + Vite + PrimeVue + Tailwind CSS + Node.js (electron-vite)
> **目标读者：** 开发团队

## 一、技术架构概览

### 1.1 整体架构

本工具采用 **Electron** 桌面框架，前端使用 **Vue 3** 构建可视化界面，后端图片压缩逻辑运行在 **Node.js** 环境中（嵌入 Electron 主进程），前后端通过 **IPC（进程间通信）** 交互。

```
┌──────────────────────────────────────────────────────┐
│                    Electron App                       │
│                                                      │
│  ┌────────────────┐         ┌────────────────────┐  │
│  │   Renderer     │  IPC    │   Main Process     │  │
│  │   (Vue 3)      │◄───────►│   (Node.js)        │  │
│  │                │         │                    │  │
│  │  - UI 界面      │         │  - 文件读写         │  │
│  │  - 图片预览     │         │  - 图片压缩引擎     │  │
│  │  - 对比视图     │         │  - 进度计算         │  │
│  │  - 参数配置     │         │  - 文件系统操作     │  │
│  └────────────────┘         └────────────────────┘  │
│                                                      │
└──────────────────────────────────────────────────────┘
```

### 1.2 技术选型说明

| 层级 | 技术选型 | 说明 |
|---|---|---|
| 桌面框架 | Electron | 成熟稳定，生态丰富，Vite 集成方案完善 |
| 构建工具 | electron-vite | 专为 Electron + Vite 设计的构建工具，开发体验好 |
| 前端框架 | Vue 3 | 轻量、响应式、组件化，与 Electron 配合良好 |
| 前端构建 | Vite | 快速的开发服务器和热更新 |
| UI 组件库 | PrimeVue | 组件丰富、样式可定制、支持 Tree-shaking |
| 样式方案 | Tailwind CSS | 原子化 CSS，快速构建自定义样式 |
| 图片压缩 | Sharp | Node.js 高性能图片处理库，支持 JPG/PNG/WebP 等格式 |
| 打包工具 | electron-builder | 将 Electron 应用打包为独立的 exe/app 可执行文件 |

### 1.3 开发与生产模式

- **开发模式**：使用 `electron-vite` 启动，开发服务器运行在前端 Renderer 进程，后端逻辑在主进程中运行，支持热更新（HMR）。
- **生产模式**：通过 `electron-builder` 打包为独立可执行文件，用户无需安装 Node.js 或任何运行时环境，双击即可运行。

### 目录结构说明

- `electron/main/`：Node.js 后端代码，运行在 Electron 主进程，负责所有底层文件操作和图片压缩逻辑。
- `electron/preload/`：预加载脚本，通过 `contextBridge` 将部分安全的 Node.js API 暴露给前端Renderer，避免直接暴露 Node.js 能力导致安全风险。
- `src/`：Vue 3 前端代码，运行在 Electron Renderer 进程，仅负责 UI 展示和用户交互，不直接操作文件系统。
- `src/components/`：纯 UI 组件，与业务逻辑解耦，可独立测试。
- `src/stores/`：Pinia 状态管理，集中管理图片列表、压缩参数、进度等全局状态。
- `src/composables/`：Vue Composables，封装可复用的业务逻辑。
- `src/api/`：封装所有与主进程的 IPC 通信调用，前端只通过此处与后端交互。

### 3.2 后端模块（Main / Node.js）

#### 3.2.1 图片压缩服务（`imageCompress.ts`）

基于 **Sharp** 库实现，是压缩功能的核心模块：

- **输入**：原始图片路径、压缩参数（质量、格式、尺寸）
- **处理流程**：
  1. 读取源图片（`sharp().metadata()` 获取尺寸、格式信息）
  2. 根据参数进行尺寸调整（如有配置）
  3. 根据输出格式和质量的参数进行压缩编码
  4. 输出到临时目录或指定路径
  5. 返回压缩结果（元数据：大小、尺寸、格式）
- **输出格式支持**：JPEG、PNG、WebP、BMP（转为上述三种之一）
- **GIF 处理**：读取首帧作为静态图处理
- **进度回调**：通过回调函数实时上报压缩进度（0-100）

#### 3.2.2 IPC 处理器

| IPC 通道 | 方向 | 说明 |
|---|---|---|
| `file:open-dialog` | Renderer → Main | 打开系统文件选择对话框 |
| `file:save-dialog` | Renderer → Main | 打开系统保存文件对话框 |
| `file:read-image-info` | Renderer → Main | 读取图片基本信息（不压缩） |
| `compress:start` | Renderer → Main | 开始压缩单张图片 |
| `compress:batch` | Renderer → Main | 开始批量压缩 |
| `compress:cancel` | Renderer → Main | 取消当前压缩任务 |
| `compress:progress` | Main → Renderer | 压缩进度实时推送 |
| `compress:result` | Main → Renderer | 压缩结果推送（含历史记录写入） |
| `file:save-to-path` | Renderer → Main | 将压缩结果保存到指定路径 |
| `history:load` | Renderer → Main | 启动时加载历史记录文件 |
| `history:add` | Renderer → Main | 追加一条历史记录 |
| `history:delete` | Renderer → Main | 删除指定历史记录 |
| `history:clear` | Renderer → Main | 清空全部历史记录 |
| `history:export` | Renderer → Main | 导出历史记录为 JSON 文件 |
| `history:import` | Renderer → Main | 导入 JSON 历史记录文件 |

### 3.3 预加载脚本（Preload）

预加载脚本通过 Electron 的 `contextBridge` 安全地暴露有限的 API 给 Renderer：

```typescript
// 仅暴露必要的操作，避免暴露完整的 Node.js API
contextBridge.exposeInMainWorld('electronAPI', {
  openFileDialog: () => ipcRenderer.invoke('file:open-dialog'),
  saveFileDialog: (options) => ipcRenderer.invoke('file:save-dialog', options),
  getImageInfo: (path) => ipcRenderer.invoke('file:read-image-info', path),
  compressImage: (options) => ipcRenderer.invoke('compress:start', options),
  compressBatch: (options) => ipcRenderer.invoke('compress:batch', options),
  cancelCompress: () => ipcRenderer.invoke('compress:cancel'),
  saveToPath: (compressedPath, targetPath) =>
    ipcRenderer.invoke('file:save-to-path', compressedPath, targetPath),
  onCompressProgress: (callback) =>
    ipcRenderer.on('compress:progress', (_, data) => callback(data)),
  onCompressResult: (callback) =>
    ipcRenderer.on('compress:result', (_, data) => callback(data)),
});
```

## 五、关键技术方案

### 5.1 图片压缩引擎

**技术选型：Sharp**

Sharp 是 Node.js 生态中性能最优的图片处理库之一，基于 libvips 构建，相比 ImageMagick 和 Jimp 在CPU和内存占用上都有显著优势。

**压缩实现要点：**

```typescript
// 伪代码描述核心逻辑
async function compress(inputPath, options) {
  let pipeline = sharp(inputPath);

  // 尺寸调整（等比缩放）
  if (options.maxWidth || options.maxHeight) {
    pipeline = pipeline.resize(options.maxWidth, options.maxHeight, {
      fit: 'inside',
      withoutEnlargement: true,
    });
  }

  // 格式与质量压缩
  const format = options.outputFormat;
  if (format === 'jpg' || format === 'jpeg') {
    pipeline = pipeline.jpeg({ quality: options.quality });
  } else if (format === 'png') {
    pipeline = pipeline.png({ quality: options.quality });
  } else if (format === 'webp') {
    pipeline = pipeline.webp({ quality: options.quality });
  } else {
    // 保持原格式
    const metadata = await sharp(inputPath).metadata();
    if (metadata.format === 'jpeg') {
      pipeline = pipeline.jpeg({ quality: options.quality });
    } else if (metadata.format === 'png') {
      pipeline = pipeline.png({ quality: options.quality });
    } else if (metadata.format === 'webp') {
      pipeline = pipeline.webp({ quality: options.quality });
    }
  }

  // 输出到临时目录
  const outputPath = path.join(tempDir, `compressed_${Date.now()}.${ext}`);
  await pipeline.toFile(outputPath);

  return outputPath;
}
```

### 5.2 进度上报机制

由于 Sharp 的压缩操作是异步的（非流式），进度无法直接从 Sharp 获取，采用以下策略模拟进度：

- 对于可预估大小的图片，按时间片估算进度（0% → 50% → 100%），前端显示动画进度。
- 压缩完成后，通过 IPC `compress:result` 将精确的文件大小、尺寸数据回传给前端。

### 5.3 图片预览与对比

- **预览**：前端通过 `<img src="file://...">` 或 Blob URL 加载本地图片文件进行展示。
- **对比视图**：采用 CSS Flexbox/Grid 布局，左右并排两张图片，使用 `transform: scale()` 实现缩放，滚动事件通过 JS 同步两个图片容器的缩放和平移状态。
- **图片数据传递**：压缩结果以临时文件路径形式存储，前端通过 `file://` 协议读取显示，无需将整个图片数据通过 IPC 传输。

### 5.4 安全模型

- Renderer 进程运行在 `contextIsolation: true` 和 `nodeIntegration: false` 模式下，无法直接访问 Node.js API。
- 所有系统级操作（文件读写、文件对话框）必须通过 `preload` 暴露的 IPC 通道调用。
- 预加载脚本仅暴露最小必要的 API 接口，不暴露完整的 `fs` 或 `process` 对象。
- 压缩后的临时文件存放在系统临时目录（如 `os.tmpdir()`），退出时清理。

### 5.5 开发调试方案

开发阶段基于 **electron-vite** 启动：

- `npm run dev`：同时启动 Vite 开发服务器（Renderer）和 Electron 主进程，支持 HMR。
- 主进程代码修改后自动重启，Renderer 代码修改后热更新。
- Renderer 中可直接通过浏览器 DevTools（右键 → 检查）进行 UI 调试。
- 主进程可通过 VS Code 的 Electron 主进程调试配置进行断点调试。

### 5.6 生产打包方案

使用 **electron-builder** 打包：

- 目标平台：Windows（exe）、macOS（app/dmg）
- 打包策略：将 Node.js 运行时、Sharp 原生模块、所有前端资源和后端代码统一打包为单一可执行文件。
- 配置文件：`electron-builder.json5`
- AutoUpdater：可预留自动更新接口（本期暂不实现）。

## 八、技术风险与应对

| 风险点 | 描述 | 应对方案 |
|---|---|---|
| Sharp 原生模块兼容性 | Sharp 需要 native 编译，在不同操作系统和 Electron 版本间可能存在 ABI 兼容问题 | 使用 `electron-rebuild` 重新编译 Sharp，或使用 `@img/sharp-libvips` 预编译二进制 |
| 大文件内存占用 | 处理超大图片（>50 MB）时可能占用大量内存 | 在前端做文件大小限制（100 MB），主进程分块读取流式处理 |
| 多平台打包差异 | Windows 和 macOS 打包命令和输出格式不同 | 通过 electron-builder 平台配置统一管理，分平台打包 |
| 图片格式支持不全 | Sharp 不直接支持 BMP/GIF 压缩 | BMP 直接转码；GIF 取首帧后按 JPEG/PNG 处理 |
| IPC 通信阻塞 | 大量图片的批量压缩可能阻塞 IPC 通道 | 使用异步 IPC invoke 配合进度回调，避免阻塞渲染线程 |
