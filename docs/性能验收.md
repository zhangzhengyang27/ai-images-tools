# 性能基准与可执行包验收

本文档用于记录图片压缩工具的性能基准、安装态 smoke 和后续优化入口。

## 验收目标

| 场景 | 验收内容 | 记录项 |
|---|---|---|
| 单张 10MB JPG | 压缩完成且 UI 有进度反馈 | 原始体积、输出体积、耗时、节省比例、内存高水位 |
| 50 张批量图片 | 批量任务逐张完成且可取消 | 总耗时、成功数、失败数、内存高水位 |
| 大文件压缩 | 处理过程不静默卡死 | 是否出现内存提示、取消是否及时生效 |
| 开发态启动 | `pnpm run dev` 可打开窗口 | 首屏时间、控制台错误 |
| unpack 产物 | `pnpm run build:unpack` 后可执行 | 启动结果、核心链路 smoke |

## 基准执行步骤

1. 准备本地样本集：
   - 1 张约 10MB 的 JPG。
   - 50 张常见尺寸图片，格式覆盖 JPG、PNG、WebP。
   - 1 张较大尺寸图片，用于验证取消和内存提示。
2. 运行开发态：
   ```bash
   pnpm run dev
   ```
3. 导入 50 张图片，确认上传区显示 `n/50` 或等价进度。
4. 执行批量压缩，记录总耗时、输出体积和底部内存高水位。
5. 使用大图开始压缩后点击取消，确认不会写入成功结果或历史记录。
6. 构建并生成 unpack 产物：
   ```bash
   pnpm run build
   pnpm run build:unpack
   ```
7. 打开 unpack 产物，重复单张导入、压缩、保存、历史记录 smoke。

## 本轮发现问题

| 优先级 | 问题 | 复现方式 | 影响 | 建议 |
|---|---|---|---|---|
| P1 | 批量压缩过程中，“保存”和“导出全部”按钮仍可点击 | 导入 8 张图片后点击“开始压缩”，当前批次尚未完成时观察右侧按钮区 | 用户可能在批量任务未完成时保存或导出半成品，产生结果不完整的误解 | 压缩期间禁用保存和导出，或在导出时明确提示仅导出已完成项 |
| P2 | 右侧“当前图片信息”和“预估压缩后大小”始终跟随当前选中图片，不跟随正在压缩的批次项 | 选中第一张后开始批量压缩，列表逐张变化，但右侧仍显示第一张的预估和详情 | 选择语义本身合理，但批量压缩时可能让用户误以为当前正在处理第一张 | 批量任务进行中增加“当前正在处理：文件名”提示，和选中图片详情区分开 |

## 后续优化入口

- 若取消仍不够及时，将 `compressImage` 移入 `worker_threads` 或独立子进程，取消时直接终止 worker/process。
- 若大批量内存高水位过高，限制批量队列并发为 1，增加更早的尺寸预估和批次建议。
- 若 unpack 产物启动慢，记录主进程启动、窗口创建和渲染首屏的分段耗时。
